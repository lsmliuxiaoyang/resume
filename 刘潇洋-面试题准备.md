## 一、Go 语言核心 ⭐ 必考 · 对应简历第 1 条

### 1. GMP 调度模型讲一下？G、M、P 分别是什么？⭐⭐⭐⭐⭐

> 简历：精通 Go 语言，深入理解 Goroutine、Channel 及 CSP 并发模型

- **G**（Goroutine）：轻量级协程，用户态线程，初始栈仅 2KB
- **M**（Machine）：OS 线程，必须绑定 P 才能执行 G
- **P**（Processor）：逻辑处理器，承载 G 的本地队列，数量 = GOMAXPROCS
- **调度流程**：新 G → P 本地队列（无锁）→ M 从 P 取 G 执行；本地空 → 从其他 P 偷一半（Work Stealing）→ 全局队列兜底
- **系统调用**：M 阻塞 → 释放 P 给其他 M → 系统调用返回后 M 尝试重新绑定 P
- **抢占调度（Go 1.14+）**：基于信号机制，避免单个 G 长期占用 CPU（时间片 10ms）

> 💡 面试杀手锏：结合简历中"Goroutine + Channel 并发优化企微好友同步，耗时由 1h+ 降至分钟级"来举例，说明不仅懂原理还会用。

### 2. Goroutine 泄漏怎么排查？⭐⭐⭐⭐⭐

> 简历：擅长 PProf 性能分析、全链路追踪及服务可观测性

- **常见原因**：channel 阻塞（发送/接收没有配对）、未取消的 context、死循环、锁未释放
- **排查工具**：`pprof` 查看 goroutine 数量和堆栈 → `go tool pprof http://localhost:6060/debug/pprof/goroutine`
- **解决**：加超时控制（context.WithTimeout）、明确退出信号（done channel）、使用协程池限制数量

> 💡 主动提到简历中写的"PProf 性能分析"，展示实操经验。面试官会追问你用 PProf 解决过什么问题。

### 3. Channel 底层数据结构是什么？发送/接收的完整流程？⭐⭐⭐⭐⭐

- **底层 hchan 结构**：环形队列（buf）+ 发送等待队列（sendq）+ 接收等待队列（recvq）+ 互斥锁（lock）+ 关闭标记（closed）
- **发送**：有等待的接收者 → 直接拷贝数据给接收者并唤醒；缓冲区未满 → 写入 buf；否则阻塞并加入 sendq
- **接收**：有等待的发送者 → 直接从发送者拷贝数据；缓冲区有数据 → 从 buf 取；否则阻塞并加入 recvq
- **关闭**：重复 close → panic；向已关闭 channel 发送 → panic；可从已关闭 channel 接收零值（用 ok 判断）

### 4. Go 内存管理：mcache / mcentral / mheap 三级分配？⭐⭐⭐⭐

> 简历：有实际 GC 优化经验

- **mcache**（P 私有）：无锁，分配小对象（<32KB），最快
- **mcentral**（全局共享）：mcache 不足时从这里申请
- **mheap**（全局堆）：管理大对象（≥32KB），向 OS 申请内存
- **分级**：Tiny 对象 <16B → 合并分配；小对象 16B~32KB；大对象 ≥32KB 直接 mheap

### 5. Go GC 三色标记法 + 混合写屏障讲一下？⭐⭐⭐⭐

> 简历：有实际 GC 优化经验

- **三色**：白色（未访问）→ 灰色（正在访问）→ 黑色（已访问完成，且子对象都处理完）
- **GC 四阶段**：STW 初始化 → 并发标记（用户协程与 GC 同时运行）→ STW 标记终止 → 并发清除
- **混合写屏障（Go 1.12+）**：并发标记期间新对象不会被误回收，无需 STW 扫描栈，停顿降至亚毫秒级
- **触发条件**：堆内存达到 GOGC（默认 100%）阈值、定时触发（2分钟）、手动 runtime.GC()

> 💡 简历中写了"GC 优化经验"，准备一个具体案例：遇到了什么问题（如 GC 频繁导致延迟抖动），怎么分析（PProf heap profile），怎么优化（减少对象分配、sync.Pool、调整 GOGC）。

### 6. defer 的执行顺序？defer 中修改命名返回值会怎样？⭐⭐⭐⭐

- **LIFO**：后进先出，defer 像压栈，执行时逆序弹出
- **参数求值时机**：defer 声明时参数立即求值（值拷贝），而非执行时
- **命名返回值**：defer 可以修改命名返回值（本质是提前声明的局部变量）
- **panic + recover**：panic 触发后逆序执行所有 defer，recover 必须在 defer 中调用才能捕获

### 7. Mutex 底层原理：正常模式 vs 饥饿模式？⭐⭐⭐⭐

- **正常模式**：等待者 FIFO 排队；新到达的 goroutine 可通过自旋尝试抢锁，可能比排队者更快获得锁
- **饥饿模式（Go 1.9+）**：等待超过 1ms 自动切换；禁止自旋，严格按 FIFO 排队，防止尾部延迟过高
- **自旋条件**：多核 CPU + 自旋次数有限 + GOMAXPROCS > 1
- **atomic vs Mutex**：atomic 基于 CPU 指令无锁无阻塞，适合简单计数/状态标记；Mutex 适合复合操作

### 8. interface 底层结构？nil 判等陷阱？⭐⭐⭐⭐

- **eface**（空接口 interface{}）：type 指针 + data 指针
- **iface**（非空接口）：itab（方法表） + data 指针
- **nil 陷阱**：接口为 nil 需要 type 和 data 都为 nil。一个 `*T` 类型的 nil 指针赋值给 interface 后，type≠nil，接口整体 ≠ nil

### 9. slice 底层原理？append 扩容规则？⭐⭐⭐

- **Header**：Data 指针 + Len + Cap，共 24 字节
- **扩容规则**：新容量 > 2倍 → 直接用新容量；否则老容量 < 256 → 翻倍；≥256 → 约 1.25 倍递增
- **注意事项**：append 可能底层数组不变（cap 够用），也可能分配新数组；多个切片共享底层数组时要小心

### 10. map 并发安全吗？怎么解决？⭐⭐⭐⭐

- **不是并发安全的**：并发读写会 panic（fatal error: concurrent map read and map write）
- **解决方案**：① sync.Mutex / sync.RWMutex 加锁保护；② sync.Map（适合读多写少、key 相对稳定）；③ 分片加锁（sharded map）
- **sync.Map 原理**：两个 map（read-only dirty），read 无锁读，写入写 dirty，达到阈值后 dirty 提升为 read

### 11. context 包的用法？取消传播机制？⭐⭐⭐⭐

- **四种创建方式**：Background / TODO / WithCancel / WithTimeout / WithDeadline / WithValue
- **取消传播**：父 context 取消 → 所有子 context 的 Done channel 自动关闭
- **使用原则**：不要存结构体里，应该作为第一个参数传递；不要传 nil；Value 只传请求范围的数据

### 12. Go 逃逸分析是什么？怎么查看？⭐⭐⭐

- **逃逸分析**：编译器决定变量分配在栈还是堆。原则：如果变量在函数返回后仍被引用，就逃逸到堆
- **查看方式**：`go build -gcflags="-m"` 查看逃逸报告
- **常见逃逸场景**：返回局部变量指针、interface 参数、闭包引用、大对象（>64KB 直接堆分配）

## 二、微服务架构 & 框架 ⭐ 对应简历第 2、5、6 条

### 13. 你们腾讯体育 App 接入层微服务是怎么拆分的？依据什么？⭐⭐⭐⭐⭐

> 简历核心亮点：主导腾讯体育 App 接入层微服务重构，QPS 提升 3 倍，接口耗时降低 66%-81%

- **拆分原则**：按业务域拆分（赛事、直播、资讯、用户等独立部署）、按变更频率拆分、单一职责
- **你的实际做法**：将单体接入层项目中的核心业务（如赛事数据、直播状态、用户信息）拆分为独立微服务，各自维护独立 DB/缓存，通过 gRPC 通信
- **拆分收益**：部署效率提升、故障隔离、资源利用率优化

> 💡 这一定是面试重头戏。准备详细的技术决策：为什么选 gRPC 而不是 HTTP？拆分过程中遇到了什么问题？灰度发布怎么做的？零故障怎么验证的？

### 14. gRPC 和 HTTP API 的区别？为什么微服务内部用 gRPC？⭐⭐⭐⭐

- **协议**：gRPC 基于 HTTP/2 + Protobuf 二进制序列化；HTTP API 基于 HTTP/1.1 + JSON 文本
- **性能**：Protobuf 体积更小、序列化更快；HTTP/2 多路复用 + 长连接 + 流式传输
- **强契约**：IDL 定义接口，避免沟通歧义
- **使用场景**：微服务内部调用 → gRPC；对外网关/前端 → HTTP

### 15. etcd 服务注册与发现的原理？挂了怎么办？⭐⭐⭐⭐

- **注册**：服务启动时把 IP+Port+服务名写入 etcd，通过心跳（lease 续约）维持存活
- **发现**：调用方 watch 服务列表变更，本地缓存一份，即使 etcd 挂了也能用本地缓存继续调用
- **一致性**：etcd 基于 Raft 协议保证强一致性
- **Kratos 中的用法**：通过 etcd registry 插件，Kratos 自动完成注册和反注册

### 16. Kratos 框架的核心特性？和 Gin/Go-zero 的区别？⭐⭐⭐⭐

> 简历：熟练使用 Kratos 框架（好未来项目）

- **Kratos**：B 站出品，面向微服务的完整框架。内置服务注册发现、配置中心、限流熔断、链路追踪、错误处理等全套组件
- **特色**：Layout 规范（整洁架构）、Wire 依赖注入、Middleware 设计、protobuf 定义 API
- **Gin**：轻量 HTTP 框架，适合 API 网关/Web 服务，生态最成熟
- **Go-zero**：好未来也有团队在用，含 goctl 代码生成工具，内置限流、熔断、链路追踪

> 💡 面试官可能会问：你在好未来为什么选 Kratos？和 Go-zero 做过对比吗？

### 17. singleflight 是什么？你们怎么用的？⭐⭐⭐⭐⭐

> 简历：singleflight 机制防止瞬时流量缓存击穿

- **原理**：对于同一个 key 的并发请求，只有一个真正执行，其他请求等待并共享结果
- **Go 标准库**：`golang.org/x/sync/singleflight`
- **使用场景**：缓存失效瞬间大量请求涌入 → singleflight 保证只有一个去查 DB，其余等待结果
- **腾讯体育实际**：赛事数据缓存过期后，大量用户同时请求 → singleflight 合并请求，防止 DB 被打垮
- **注意事项**：不适合耗时差异大的场景（一个慢请求拖慢所有等待者）；需要设置超时

### 18. 限流、熔断、降级的区别？Go 中怎么实现？⭐⭐⭐⭐

- **限流**：保护自己不被流量打垮。算法：令牌桶（允许突发）/ 漏桶（平滑流量）/ 滑动窗口
- **熔断**：防止级联故障。状态：关闭→打开→半开。触发条件：错误率高/慢调用多
- **降级**：主动放弃非核心功能保核心链路
- **Go 实现**：sentinel-golang / hystrix-go / Kratos 内置中间件

### 19. 服务优雅启停怎么实现？发布不丢请求？⭐⭐⭐⭐

- 监听 SIGINT/SIGTERM 信号
- 先将健康检查置为不健康（注册中心摘除流量）
- 停止接收新请求，等待已有请求处理完（graceful period 通常 30s）
- 关闭 DB/Redis/MQ 连接池，退出进程
- Kratos 框架内置了这套机制

### 20. 链路追踪怎么做？线上接口慢怎么排查？⭐⭐⭐⭐

> 简历：接入天机阁全链路追踪

- **核心三元组**：traceId（请求链唯一标识）+ spanId（每段操作）+ parentSpanId
- **排查步骤**：① 看监控大盘（CPU/内存/GC/Goroutine 数）→ ② 看链路追踪（哪段耗时最高）→ ③ 看具体段：DB 慢 → EXPLAIN 分析 SQL；RPC 慢 → 看对端；自身慢 → PProf CPU profile
- **天机阁**：腾讯内部链路追踪系统，类似开源的 Jaeger/SkyWalking

### 21. Docker + CI/CD 持续集成的实践？⭐⭐⭐

> 简历：有 CI/CD 持续集成与持续部署实践经验

- **流程**：代码提交 → 自动构建 → 运行单测（覆盖率检查）→ 构建 Docker 镜像 → 推送镜像仓库 → 部署到测试/预发布/生产环境
- **关键**：多阶段构建减小镜像体积、环境变量管理、健康检查配置

### 22. gRPC 超时怎么控制？context 怎么传递？⭐⭐⭐

- 每层必须设置超时，不设超时会导致雪崩
- 使用 `context.WithTimeout` 传递截止时间，gRPC 会自动将 deadline 通过 metadata 传递给下游
- Kratos 框架内置了 RPC 超时配置，可以在配置文件中设置

## 三、MySQL 深度 ⭐ 对应简历第 3 条

### 23. MySQL 索引底层数据结构？为什么用 B+ 树？⭐⭐⭐⭐⭐

- B+ 树非叶子节点只存索引 key，叶子节点存完整数据并用双向链表连接
- **优势**：树高度低（IO 次数少）、叶子链表支持范围查询、查询效率稳定（O(log n)）
- **对比 B 树**：B 树非叶子也存数据 → 每个节点能存的 key 更少 → 树更高 → IO 更多
- **对比 Hash**：Hash 等值查询 O(1) 但无法范围查询、无法排序

### 24. 聚簇索引和非聚簇索引的区别？回表是什么？⭐⭐⭐⭐⭐

- **聚簇索引**（InnoDB 主键索引）：叶子节点存储完整行数据；一张表只有一个
- **非聚簇索引**（二级索引）：叶子节点存储主键值；查询需要回表取完整数据
- **覆盖索引**：查询列都在二级索引中 → 不需要回表，性能最优
- **回表**：通过二级索引找到主键 → 再到聚簇索引查完整行，多一次 IO

### 25. 你是怎么做慢查询分析和优化的？⭐⭐⭐⭐⭐

> 简历：丰富的慢查询分析与性能调优经验

- **定位**：开启 slow_query_log，设置 long_query_time；或通过监控平台（Prometheus + Grafana）
- **分析**：EXPLAIN 看执行计划（type 级别从 system→const→eq_ref→ref→range→index→ALL，越靠前越好）；关注 key（使用索引）、rows（扫描行数）、Extra（Using filesort/Using temporary 是坏信号）
- **优化手段**：① 加索引（最左前缀原则）② 覆盖索引避免回表 ③ SQL 改写（避免 SELECT *、避免函数作用索引列）④ 分库分表 ⑤ 读写分离

> 💡 准备一个你在腾讯体育或好未来实际处理过的慢查询案例。

### 26. MySQL 事务隔离级别？每个级别解决了什么问题？⭐⭐⭐⭐

- **读未提交**：脏读（读到未提交的数据）
- **读已提交**（RC）：解决脏读，但不可重复读（同事务内两次读结果不同）
- **可重复读**（RR，InnoDB 默认）：解决不可重复读，但可能幻读（InnoDB 通过 Next-Key Lock 部分解决）
- **串行化**：解决全部问题，但性能最差

### 27. InnoDB 的行锁有哪些？间隙锁是什么？⭐⭐⭐⭐

- **Record Lock**：锁单行记录
- **Gap Lock**：锁索引间隙，防止插入幻读
- **Next-Key Lock**：Record Lock + Gap Lock，左开右闭区间
- **死锁**：两个事务互相等待对方的锁。解决方案：按相同顺序加锁、死锁检测（innodb_deadlock_detect）、超时回滚

### 28. MVCC 原理？ReadView 是什么？⭐⭐⭐⭐

- **MVCC**：多版本并发控制，通过 undo log 保留数据的历史版本，实现非锁定读
- **实现**：每行记录有 DB_TRX_ID（最近修改的事务ID）和 DB_ROLL_PTR（undo log 指针）
- **ReadView**：快照读时创建，记录当前活跃事务列表。根据可见性规则判断某行数据是否对当前事务可见
- **RC vs RR**：RC 每次 SELECT 创建新 ReadView；RR 事务中第一次 SELECT 创建 ReadView，后续复用

### 29. 最左前缀原则是什么？索引失效的场景有哪些？⭐⭐⭐⭐

- **最左前缀**：联合索引 (a,b,c)，查询条件从最左列开始匹配：a 可以用索引、a+b 可以、a+c 只能用 a 列、b+c 不能用
- **失效场景**：① LIKE '值%' 可以用，LIKE '%值' 失效 ② 索引列上做函数/计算 ③ 隐式类型转换 ④ OR 条件中有非索引列 ⑤ NOT IN / != / <>（可能失效）

### 30. 分库分表方案？怎么做平滑迁移？⭐⭐⭐

- **垂直拆分**：按业务模块拆（用户库、订单库）
- **水平拆分**：按某个字段 hash 或 range 拆分（如按 user_id % 64）
- **迁移策略**：双写 → 历史数据迁移 → 读流量切换 → 停旧库写入 → 清理

## 四、Redis 深度 ⭐ 对应简历第 4 条

### 31. 缓存穿透、击穿、雪崩的区别和解决方案？⭐⭐⭐⭐⭐

> 简历：有高并发缓存架构设计经验

- **穿透**：查不存在的数据（恶意攻击）→ 布隆过滤器 / 缓存空值 + 短过期
- **击穿**：热点 key 过期，大量请求打到 DB → singleflight 合并请求 / 互斥锁 / 永不过期 + 异步更新
- **雪崩**：大量 key 同时过期 / Redis 宕机 → 过期时间加随机值 / 多级缓存 / 限流降级 / Redis 集群高可用

> 💡 简历中写到 singleflight 防止缓存击穿，这是你的优势！讲清楚你在腾讯体育赛事系统中是怎么用的。

### 32. Redis 五种数据类型及应用场景？⭐⭐⭐

- **String**：缓存、计数器、分布式锁、Session
- **Hash**：对象存储（用户信息）、购物车
- **List**：消息队列、时间线
- **Set**：标签、共同好友、去重
- **ZSet**：排行榜、延迟队列、带权重的集合

### 33. Redis 持久化 RDB 和 AOF 的区别？⭐⭐⭐⭐

- **RDB**：快照，定时全量备份。恢复快，但可能丢失最后一次快照后的数据。fork 子进程 dump，不影响主进程
- **AOF**：追加写日志，记录每条写命令。数据更安全（可配置 everysec），但文件更大、恢复更慢。AOF 重写压缩
- **混合持久化（Redis 4.0+）**：RDB 快照 + AOF 增量，结合两者优势

### 34. Redis 过期策略和内存淘汰策略？⭐⭐⭐⭐

- **过期策略**：惰性删除（访问时检查）+ 定期删除（每 100ms 随机抽取一批检查）
- **内存淘汰（8种）**：noeviction（不淘汰，拒绝写入）/ allkeys-lru / volatile-lru / allkeys-lfu / volatile-lfu / allkeys-random / volatile-random / volatile-ttl
- **LRU vs LFU**：LRU 最近最少使用（访问时间）；LFU 最不经常使用（访问频率），适合有冷热数据的场景

### 35. Redis 大 Key 怎么发现和治理？⭐⭐⭐⭐

> 简历：Redis 大 Key 拆分治理

- **发现**：`redis-cli --bigkeys` 扫描；`MEMORY USAGE key` 查看内存占用；RDB 分析工具（redis-rdb-tools）
- **危害**：阻塞 Redis（DEL 大 Key 会卡主线程）、内存不均、网络带宽打满
- **治理**：① 拆分（大 Hash 按 field 拆分；大 List 分桶）② 压缩 value ③ 改用其他存储（如 ES）④ 异步删除（UNLINK）

> 💡 准备你在腾讯体育赛事系统 Redis 大 Key 拆分的具体案例。

### 36. 分布式锁怎么实现？Redlock 原理？⭐⭐⭐⭐

- **单节点**：`SET lock_key value NX PX 30000`（原子加锁+设置过期）
- **解锁**：Lua 脚本先判断 value（自己的唯一标识）再删除，保证原子性
- **Redlock**：多节点加锁，过半成功才算成功。解决单点故障问题
- **注意**：锁过期时间要大于业务执行时间；考虑续期机制（Watch Dog）

## 五、Kafka 消息队列 ⭐ 对应简历第 5 条

### 37. Kafka 为什么这么快？⭐⭐⭐⭐⭐

- **顺序写磁盘**：追加写入，速度可达 600MB/s，接近内存速度
- **零拷贝（sendfile）**：数据从磁盘 → 内核缓冲区 → 网卡，不经过用户态，减少 2 次拷贝
- **批量+压缩**：生产者攒批发送，减少网络往返
- **分区并行**：一个 Topic 多分区，消费者并行消费，吞吐量线性增长
- **Page Cache**：利用 OS 的页缓存，热数据直接内存读

### 38. 如何保证消息不丢失？⭐⭐⭐⭐⭐

- **生产者**：acks=all（所有 ISR 副本确认）+ retries 重试
- **Broker**：多副本（replication.factor ≥ 3）+ min.insync.replicas ≥ 2 + unclean.leader.election.enable=false
- **消费者**：关闭自动提交 offset，业务处理完再手动提交；先处理再提交（at-least-once + 业务幂等）

### 39. 消息重复消费怎么解决？（幂等性）⭐⭐⭐⭐⭐

- **根本原则**：MQ 保证 at-least-once（至少一次），业务层自己保证幂等
- **方案**：① 数据库唯一约束（insert 去重）② Redis 记录已处理的消息 ID ③ 乐观锁（version 字段）④ 分布式锁
- **你简历中的实践**：Redis 记录订单游标 → 防止重复处理与丢失风险，这就是幂等性保证

### 40. 如何保证消息顺序性？⭐⭐⭐⭐

- **核心思路**：局部串行而非全局串行。相同业务 key 的消息路由到同一个分区
- **生产端**：指定 key 做 hash 路由到同一个分区；开启幂等生产者防止重试导致乱序
- **消费端**：同一个分区单线程消费
- **Kafka**：max.in.flight.requests.per.connection=1 或 enable.idempotence=true

### 41. 消息积压怎么办？⭐⭐⭐⭐

- **临时扩容**：增加消费者数量（但不能超过分区数）+ 增加分区
- **海量积压**：创建新 Topic（分区数翻多倍），写搬运程序把积压消息转发到新 Topic（只转发不处理业务），部署大量消费者快速消费
- **治本**：排查消费慢的原因（下游 DB 慢、第三方接口超时等）

### 42. Kafka ISR 机制是什么？acks 参数含义？⭐⭐⭐

- **ISR**：In-Sync Replicas，与 Leader 保持同步的副本集合
- **acks=0**：不等待确认，可能丢失（最快）
- **acks=1**：Leader 写入即确认（Leader 宕机可能丢失）
- **acks=all/-1**：所有 ISR 副本确认（最安全，最慢）

### 43. Kafka Consumer Group 重平衡（Rebalance）机制？⭐⭐⭐

- **触发条件**：消费者加入/离开、分区数变化、心跳超时
- **过程**：所有消费者停止消费 → 重新分配分区 → 恢复消费（期间有 STW）
- **优化**：增大 session.timeout.ms 和 heartbeat.interval.ms；使用 CooperativeStickyAssignor（增量重平衡，减少影响）

### 44. 你在项目里怎么用 Kafka 的？⭐⭐⭐⭐⭐

> 简历多处提到 Kafka

- **好未来广告投放系统**：广告点击事件 → Kafka → 实时处理（归因、去重、转化计算）
- **腾讯体育赛事系统**：赛事数据通过 Kafka 发布/订阅，保障数据有序分发不丢失
- **私域运营平台**：抖店订单变更 → Kafka → 消费者处理（课程开通、企微同步）

### 68. 单元测试怎么保证 90%+ 的覆盖率？写了哪些类型的测试？⭐⭐⭐⭐

- **分层测试**：单元测试（核心逻辑）+ 集成测试（接口级）+ 端到端测试（关键链路）
- **工具**：Go testing + testify + gomock（mock 依赖）；go test -cover 检查覆盖率
- **策略**：增量代码必须 90%+ 覆盖（CI 检查）；存量代码逐步补充
- **表驱动测试**：Go 推荐方式，一组输入输出用表格驱动

### 69. 你们团队的代码审查规范是怎么样的？⭐⭐⭐

- **工具**：golangci-lint 自动检查 + CI 拦截
- **流程**：PR → AI 先过一遍（基础问题）→ 至少 1 人 Review → 所有评论解决 → 合并
- **Review 关注点**：功能正确性、性能、安全、可读性、错误处理、并发安全

### 70. CI/CD 流水线怎么设计的？⭐⭐⭐

- **CI**：代码提交 → lint 检查 → 单元测试（覆盖率检查）→ 构建 → 集成测试 → 安全扫描
- **CD**：构建 Docker 镜像 → 推送镜像仓库 → 部署测试环境 → 自动化测试 → 预发布 → 灰度发布 → 全量
- **工具链**：GitLab CI / Jenkins / GitHub Actions + Docker + K8s
